# Recallr — shadcn/ui + Tailwind Migration Design

**Date:** 2026-06-21 (rev. 2 — dark/light theme added)
**Status:** Approved (pending)
**Scope:** All three UI surfaces (popup, content script panel, dead-code cleanup)

## Context

Recallr is a Manifest V3 Chrome extension (React 19, Zustand 5, Vite 8, `@crxjs/vite-plugin`). It has three React roots: a 320px toolbar popup, a long-lived content script that mounts a shadow DOM and currently renders nothing, and a service worker (no UI). The popup has four screens (Login, Dashboard, Save, Chat) plus a shell.

Styling is currently hand-rolled: CSS custom properties in `src/styles/tokens.css` (editorial paper palette — cream, terracotta, Newsreader serif) consumed by ~600 lines of CSS Modules (`.module.css` per component). There is no Tailwind, no PostCSS config, no shadcn/ui, no Radix primitives, no lucide-react, no clsx/tailwind-merge/cva. The "editorial paper" look is dropped entirely.

**Two structural issues this design addresses:**

1. **Inconsistent component primitives.** Forms, buttons, and message bubbles are hand-rolled per screen. The `ChatInput` and `MessageList` are the only things shared across surfaces; everything else is duplicated.
2. **Content script panel is dead code.** `ChatPanel`, `panelStore`, and `usePanel` exist but are never mounted. `ContentRoot` returns null. `LoggedInScreen.tsx` is also orphaned — `App.tsx` routes to `DashboardScreen` instead.

**Intended outcome:** All UI surfaces use shadcn/ui primitives on a Tailwind v4 base, themed with shadcn's **default Neutral theme** in **both light and dark mode**, with a working toggle. Shared form/feedback primitives replace hand-rolled markup. CSS Modules are deleted. The content script's `ChatPanel` is wired up and migrated. Dead code is removed.

## Decisions (locked from brainstorming)

- Adopt Tailwind v4 + shadcn/ui fully. No coexistence period, no partial migration.
- **Aesthetic: shadcn's default theme — Neutral base color, light and dark mode both supported, with a toggle.** (Previous rev. 1 of this doc proposed a static zinc/slate light-only theme; superseded.) The terracotta accent and serif wordmark are dropped; "Recallr" wordmark in `App.tsx` becomes the only branded survivor.
- Scope: all four popup screens + the content script's `ChatPanel` (now wired) + dead-code cleanup + theme system.
- Primitives to install: Button, Input, Label, Textarea, Alert, Badge, ScrollArea, Card, DropdownMenu, Tooltip, Separator, Skeleton. (The "all essentials" bundle.)
- Migration style: one-shot rewrite. Delete `.module.css` files as their `.tsx` is rewritten. No parallel systems.
- Theme is a **manual two-state toggle** (light / dark), not OS-level `prefers-color-scheme` auto-detection. No "system" option in this pass — see Out of Scope.

## Architecture

### Tooling

| Concern | Before | After |
|---|---|---|
| CSS pipeline | Vite's built-in CSS Modules | Tailwind v4 via `@tailwindcss/vite` plugin |
| Style authoring | Hand-written `.module.css` per component | Tailwind utility classes inline on JSX |
| Component primitives | Raw `<button>`, `<input>`, hand-rolled styles | shadcn/ui components in `src/components/ui/` |
| Class composition | Template strings, manual clsx | `cn()` helper from `src/lib/utils.ts` (clsx + tailwind-merge) |
| Variant APIs | None | `class-variance-authority` for compound components |
| Icons | None (× glyph inline) | `lucide-react` (X, ArrowLeft, LogOut, Send, Sun, Moon, etc.) |
| Path alias | None (relative imports) | `@/*` → `src/*` in `tsconfig.app.json` + `vite.config.ts` |
| Theme | CSS variables in `src/styles/tokens.css`, light-only | CSS variables in `src/index.css` (Tailwind v4 `@theme` block), **`:root` (light) + `.dark` (dark) variants**, generated by shadcn CLI with Neutral base |
| Theme persistence | N/A | `chrome.storage.local`, read on boot by a `ThemeProvider`, synced across popup/content/background |

### Path alias

`tsconfig.app.json` adds:
```json
"baseUrl": ".",
"paths": { "@/*": ["src/*"] }
```
`vite.config.ts` adds a matching `resolve.alias` so `@/components/ui/button` resolves correctly. This unblocks the shadcn CLI.

### Theme & dark mode

This is the part that changed from rev. 1, and it's the part with the most moving pieces, so it gets its own section.

**Generation.** Run `npx shadcn@latest init`, select **Neutral** as the base color, and accept the default (it scaffolds both `:root` and `.dark` variable blocks into `src/index.css` automatically — exact token values aren't hand-authored, they come from the CLI). The shape looks like:

```css
@import "tailwindcss";

:root {
  --background: ...;
  --foreground: ...;
  --primary: ...;
  --primary-foreground: ...;
  --secondary: ...;
  --secondary-foreground: ...;
  --muted: ...;
  --muted-foreground: ...;
  --accent: ...;
  --accent-foreground: ...;
  --destructive: ...;
  --border: ...;
  --input: ...;
  --ring: ...;
  --radius: 0.625rem;
  /* + card, popover, sidebar, chart variants per shadcn's full default set */
}

.dark {
  --background: ...;
  --foreground: ...;
  /* same key set, dark values */
}

@theme inline {
  --color-background: var(--background);
  --color-foreground: var(--foreground);
  /* ...maps every CSS var into a Tailwind color token */
}
```

If a different base color (Slate, Zinc, Gray, Stone) is preferred later, re-run `init` or swap the block — this is a 10-minute change since no component code references raw hex values.

**Where `.dark` gets applied.** shadcn's dark mode works by toggling a `dark` class on an ancestor element (normally `<html>`). Recallr has three independent DOM roots that don't share an ancestor:

| Root | Where `.dark` is applied |
|---|---|
| Popup | `<html>` in `popup.html` / `document.documentElement` in `main.tsx` |
| Content script panel | The **shadow root's host wrapper element** (the div `mount.ts` creates before attaching shadow DOM), since `.dark` on the real page's `<html>` is irrelevant inside a shadow root |
| Service worker | N/A — no UI |

**`ThemeProvider` (new — `src/lib/theme-provider.tsx`).** A small context provider, not `next-themes` (this isn't Next.js):
- State: `theme: 'light' | 'dark'`.
- On mount: reads `chrome.storage.local.get('theme')`; defaults to `'light'` if unset.
- On change: writes to `chrome.storage.local.set({ theme })` and toggles the `dark` class on the relevant root element (passed in as a prop/ref, since popup and content script attach it to different elements).
- Subscribes to `chrome.storage.onChanged` so that toggling in the popup is reflected in an already-open `ChatPanel` (and vice versa) without a reload.

**Why `chrome.storage.local` and not `localStorage`:** the content script's `localStorage` would resolve to the *host page's* origin, not the extension's — it would not be visible to the popup or service worker, and would leak into whatever site the user is on. `chrome.storage.local` is scoped to the extension and accessible from all three contexts. This replaces the simpler `localStorage`-based pattern most shadcn dark-mode tutorials use.

**`ThemeToggle` (new — `src/components/ThemeToggle.tsx`).** A single icon `Button` (`variant="ghost" size="icon"`) showing `Sun` or `Moon` (lucide-react) depending on current theme, calling `toggleTheme()` from `ThemeProvider`'s context. Mounted once, in the popup's `App.tsx` header, next to the wordmark. The content script's `ChatPanel` does **not** get its own toggle in this pass — it inherits whatever the popup last set, via the `chrome.storage.onChanged` listener. (See Out of Scope.)

## Component mapping (content script)

- `mount.ts` adds a step to inject the compiled `index.css` into the shadow root (alongside the Google Fonts `<link>`, which is removed since we use system fonts), and creates the wrapper element that `ThemeProvider` toggles `.dark` on.
- `root.tsx` becomes a real component: reads `usePanelStore.open`, wraps in `<ThemeProvider>`, renders `<ChatPanel />` when set, else `null`. The floating launcher (a 56px button in the bottom-right of the page) is a separate new file `src/content/FloatingTrigger.tsx` that calls `usePanelStore.show('radial')` on click and is always rendered at the top of `ContentRoot`.
- `ChatPanel` becomes a shadcn `<Card>` with `className="fixed bottom-4 right-4 w-[400px] h-[540px] shadow-2xl"` (same dimensions as before, but the slide-up animation is preserved via Tailwind `data-[state=open]:animate-in slide-in-from-bottom-4`).
- The radial menu (`'radial' | 'save' | 'chat'`) is not built in this migration — `panelStore` retains the type but `FloatingTrigger` only opens `'chat'` for now.

## Dead code removed

- `src/popup/screens/LoggedInScreen.tsx` — orphaned; `App.tsx` never references it.
- `src/hooks/usePanel.ts` — never imported; `ChatPanel` uses `usePanelStore` directly. Delete.
- `src/store/panelStore.ts` — kept because `FloatingTrigger` and `ChatPanel` both consume it.

## Configuration files

- `tsconfig.app.json`: add `baseUrl` + `paths` for `@/*`.
- `vite.config.ts`: add `tailwindcss()` from `@tailwindcss/vite`; add `resolve.alias` for `@`.
- `package.json` adds: `tailwindcss@^4`, `@tailwindcss/vite`, `class-variance-authority`, `clsx`, `tailwind-merge`, `lucide-react`, `@radix-ui/react-slot`, `@radix-ui/react-label`, `@radix-ui/react-dropdown-menu`, `@radix-ui/react-tooltip`, `@radix-ui/react-separator`, `@radix-ui/react-scroll-area`, `@radix-ui/react-dialog`. No extra dependency is needed for the theme toggle — it's a plain Button + context, not a Radix primitive.
- `manifest.json`: confirm `"storage"` permission is present (required for `chrome.storage.local`; likely already there for auth token storage, but verify).
- `components.json` (new) at the repo root: standard shadcn config, base color `neutral`, `cssVariables: true`, paths set to `@/components`, `@/lib/utils`, `@/components/ui`.

## Data flow

No changes except the new theme channel. `authStore`, `chatStore`, `panelStore`, `popupStore`, messaging, storage, `background/index.ts` all stay exactly as they are. `useAuth`, `useChat` hooks are unchanged. The only new piece of state is `theme`, owned by `ThemeProvider` and persisted via `chrome.storage.local`, independent of the existing stores. `verbatimModuleSyntax` stays on, so any type-only shadcn imports use `import type`.

## Error handling

No changes. `messaging` still wraps `chrome.runtime.lastError` and rejects on failure. `chatStore.send` still replaces the loading bubble with an error fallback, now rendered as a shadcn `<Alert variant="destructive">`. If `chrome.storage.local.get('theme')` fails or is unavailable, `ThemeProvider` falls back to `'light'` silently rather than throwing.

## Testing

No test framework is added. Verification is manual:

1. **Build check:** `npm run build` must succeed (runs `tsc -b && vite build`).
2. **Dev loop:** `npm run dev` must serve without console errors.
3. **Popup flow:** Load unpacked in Chrome → click toolbar icon → log in (`demo`/`demo`) → see Dashboard → click "Save URL" → save current page → see success Alert → click "Chat" → send a message → see user + AI bubbles render.
4. **Content script flow:** With the dev build loaded, navigate to any page → see floating launcher button (bottom-right) → click it → see ChatPanel slide up → type a question → see response → close panel.
5. **Theme flow (new):** In the popup, click the Sun/Moon toggle → confirm immediate light↔dark switch with no flash of unstyled content → close and reopen the popup → confirm it remembers the chosen theme → open the content script panel → confirm it matches the popup's last-set theme → toggle again from the popup while the panel is still open → confirm the panel updates live.
6. **Visual check:** Confirm shadcn Neutral palette in both modes (light: white/near-white background, dark: near-black background, no cream, no terracotta), system font, no paper-grain.
7. **Console:** No React warnings, no missing CSS, no 404s on the injected shadow CSS link, no `chrome.storage` permission errors.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Tailwind v4 + shadcn requires `@theme` block in the right order | Follow shadcn's v4 init docs exactly; `npx shadcn@latest init` generates the right starter |
| Shadow DOM does not inherit page styles, including Tailwind utilities | `mount.ts` injects the compiled CSS into the shadow root via `<link>` to the built asset URL |
| `verbatimModuleSyntax` will reject type-only Radix imports without `import type` | All shadcn-generated files already use this pattern; verify after init |
| `all: initial` on the shadow host will break Tailwind preflight | Only the host's outer frame is reset; the inner mount is normal CSS |
| **Three isolated JS contexts (popup, content script, service worker) don't share `localStorage`** | Use `chrome.storage.local` for theme persistence instead, with `chrome.storage.onChanged` to sync live |
| **Flash of wrong theme on popup open** (theme read is async via `chrome.storage.local.get`) | Render a neutral skeleton/blank shell until the first `get` resolves (it's fast, but not synchronous like `localStorage`), or cache last-known theme in a tiny synchronous fallback |
| **`.dark` class has no natural ancestor across the shadow root boundary** | `mount.ts`'s host wrapper element is the explicit attach point for `.dark`, applied by `ThemeProvider`, not `document.documentElement` |
| Font regression: the popup's wordmark used Newsreader, now system font | Intentional per the design decision — call it out in the PR description |
| `manifest.json` missing `"storage"` permission | Verify before implementation; add if absent (one-line manifest change) |
| shadcn primitives copy-pasted into the repo, future updates are manual | Standard shadcn trade-off; document in PR description |

## Out of scope

- A new options page or side panel surface.
- A radial menu (the `'radial'` kind in `panelStore` is reserved but not built).
- The `'save'` kind in `panelStore` (a floating save panel separate from the popup).
- **OS-level `prefers-color-scheme` auto-detection / a "system" theme option.** This pass is an explicit manual two-state toggle only.
- A separate theme toggle inside `ChatPanel` — it inherits the popup's last-set preference.
- Test framework setup.
- Building a `FloatingTrigger` that opens the radial menu (only chat is wired).
- Migrating the `background/index.ts` service worker (no UI).
